Інформаційне та програмне забезпечення віртуального тренажеру для дистанційного курсу “Системний аналіз”

З дисципліни: «Управління IT проєктами»

Метою роботи є проєктування інформаційного та програмного забезпечення віртуального тренажеру для дистанційного курсу «Системний аналіз», що включає розробку архітектури системи, бази даних, модулів навчальних сценаріїв та алгоритмів автоматизованого оцінювання знань студентів.

Архітектура та інформаційна модель системи
Проєктування архітектури є основоположним етапом, що визначає технічну реалізацію тренажеру на всіх рівнях: від фізичного розміщення компонентів до логіки взаємодії між модулями. Для забезпечення надійності, масштабованості та зручності підтримки обрано клієнт-серверну трирівневу архітектуру (Three-tier architecture).
Рівень 1 — Рівень представлення (Presentation Tier): клієнтська частина (frontend), реалізована мовами HTML5, CSS3 та JavaScript із використанням бібліотеки React. Цей рівень відповідає за відображення інтерфейсу користувача, збір введених даних та їх передачу на сервер. Використання React забезпечує побудову динамічних компонентів без перезавантаження сторінки, що критично важливо для інтерактивних навчальних вправ.
Рівень 2 — Рівень бізнес-логіки (Logic Tier): серверна частина (backend), реалізована мовою Python із використанням вебфреймворку Django та Django REST Framework. На цьому рівні виконуються всі алгоритми перевірки відповідей студентів, розрахунки показників успішності, генерація звітів та управління навчальним процесом. Вибір Django обґрунтований наявністю потужної вбудованої системи автентифікації, ORM для роботи з базою даних та зрілої екосистеми пакетів.
Рівень 3 — Рівень даних (Data Tier): реляційна база даних PostgreSQL, що зберігає всю інформацію про студентів, навчальний контент, результати виконання завдань та налаштування системи. PostgreSQL обрана за надійність, підтримку складних запитів та транзакційну цілісність.
Для моделювання функцій системи застосовано методологію IDEF0. На рівні контекстної діаграми (A-0) головна функція визначається як «Організація інтерактивного навчання системному аналізу в дистанційному форматі».

Основні компоненти моделі IDEF0:
Вхідні дані (Input): дані студентів (облікові записи), навчальний контент (теорія, задачі), відповіді студентів на виконані вправи.
Вихідні дані (Output): перевірені відповіді з педагогічним коментарем, звіти про навчальний прогрес, аналітика для викладача щодо засвоєння матеріалу.
Управління (Control): навчальна програма дисципліни, методичні рекомендації, критерії оцінювання знань.
Механізми (Mechanism): вебсервер Django, база даних PostgreSQL, браузер студента, адміністративна панель викладача.
Декомпозиція головної функції (діаграма A0) виділяє такі підфункції: реєстрація та управління користувачами; перегляд теоретичного матеріалу; виконання інтерактивних вправ; автоматична перевірка відповідей; формування звітності.
Для об'єктно-орієнтованого опису системи розроблено комплекс UML-діаграм. Діаграма прецедентів (Use Case Diagram) визначає три основні ролі:
1. Студент: проходить реєстрацію, вивчає теоретичний матеріал, виконує практичні завдання, переглядає результати та власний прогрес.
2. Викладач: управляє навчальним контентом, переглядає статистику успішності групи, встановлює терміни виконання модулів, додає нові завдання та редагує еталонні рішення.
3. Адміністратор: управляє обліковими записами користувачів, налаштовує параметри системи, моніторить технічний стан серверної частини та виконує резервне копіювання бази даних.
Діаграма послідовності (Sequence Diagram) описує ключовий сценарій взаємодії: студент через браузер вибирає задачу з теми «Метод аналізу ієрархій», вводить матрицю попарного порівняння, надсилає запит через REST API до сервера Django, який виконує алгоритм перевірки узгодженості суджень, повертає результат перевірки та педагогічний коментар до браузера, де вони відображаються студенту.
2.2. Проєктування бази даних тренажеру
База даних тренажеру є центральним елементом системи, що зберігає як навчальний контент, так і результати роботи студентів. Правильна структура бази даних безпосередньо впливає на продуктивність системи та гнучкість її масштабування. Для реалізації обрано реляційну СУБД PostgreSQL, яка забезпечує надійне зберігання даних, підтримку складних запитів з агрегацією та механізми транзакційної цілісності.
На основі аналізу предметної області та функціональних вимог визначено такі основні інформаційні сутності:
Користувачі (Users): зберігає дані облікових записів усіх учасників системи — унікальний ідентифікатор, логін, хеш паролю, електронна адреса, роль (student / teacher / admin), дата реєстрації та статус активності облікового запису.
Групи (Groups): навчальні групи студентів — ідентифікатор групи, назва групи, рік навчання, посилання на відповідального викладача. Забезпечує прив'язку студентів до академічних груп для коректної звітності.
Модулі курсу (CourseModules): тематичні блоки дисципліни — ідентифікатор, назва блоку (наприклад, «Теорія систем», «Метод аналізу ієрархій»), опис, порядковий номер у курсі, статус активності.
Теоретичні матеріали (TheoryMaterials): навчальний контент для кожного модуля — ідентифікатор, посилання на модуль, заголовок, текст у форматі HTML, вкладені файли та мультимедіа.
Завдання (Tasks): база практичних завдань — ідентифікатор, посилання на модуль, тип завдання (тест, розрахункова задача, задача на побудову схеми), умова задачі, рівень складності (1–3), еталонна відповідь (у JSON-форматі для розрахункових задач або у вигляді ключів для тестів), пояснення до правильного рішення.
Результати виконання (TaskResults): зберігає відповіді студентів — ідентифікатор, посилання на студента, посилання на завдання, введена відповідь, автоматична оцінка (вірно/невірно/частково), кількість балів, час початку та завершення виконання, кількість спроб.
Прогрес по модулях (ModuleProgress): агрегований прогрес студента по кожному модулю — посилання на студента, посилання на модуль, відсоток виконаних завдань, загальна кількість балів, статус модуля (не розпочато / в процесі / завершено).
Досягнення (Achievements): система значків геймифікації — назва досягнення, умова отримання, іконка. Зв'язана з таблицею студентів через проміжну таблицю.
Ключовий зв'язок у базі даних встановлюється між сутностями «Студент» → «Результат виконання» → «Завдання» → «Модуль». Цей ланцюжок дозволяє формувати детальну аналітику: скільки спроб зробив студент для вирішення конкретного завдання, які типи задач викликають найбільші труднощі, яка динаміка засвоєння матеріалу по тижнях курсу.
Особливу роль відіграє сутність «Завдання» з полем типу відповіді. Для завдань типу «тест» еталонна відповідь зберігається як рядковий ключ. Для розрахункових задач (наприклад, обчислення вектора пріоритетів у МАІ) відповідь зберігається у форматі JSON із допустимими відхиленнями значень, що дозволяє алгоритму перевірки враховувати похибки округлення.
2.3. Програмне забезпечення та інтерфейс користувача
Програмне забезпечення тренажеру розроблено за принципом модульності, де кожен функціональний блок є самостійним та може розвиватися незалежно. Серверна частина реалізована на основі фреймворку Django (Python), клієнтська — на основі бібліотеки React (JavaScript). Взаємодія між ними здійснюється через REST API у форматі JSON.
Структура серверної частини (backend):
Серверна частина організована відповідно до архітектурного патерну MVT (Model-View-Template), адаптованого для REST API. Основні прикладні модулі:
Модуль автентифікації (auth_app): реалізує реєстрацію, вхід, вихід із системи та відновлення паролю. Використовує JWT-токени (JSON Web Tokens) для безстанової автентифікації в REST API.
Модуль курсу (course_app): управляє модулями курсу, теоретичними матеріалами та базою завдань. Надає API-ендпоінти для отримання контенту.
Модуль перевірки (checker_app): містить алгоритми автоматичної перевірки відповідей для кожного типу задач. Це ключовий модуль системи. Алгоритм перевірки задач на МАІ: приймає від студента матрицю попарного порівняння, розраховує вектор пріоритетів та індекс узгодженості (CR), порівнює з еталонним рішенням із урахуванням допустимого відхилення ±0.01.
Модуль аналітики (analytics_app): агрегує дані про прогрес студентів, формує статистичні звіти для викладача, розраховує показники ефективності навчання.
Структура клієнтської частини (frontend):
Клієнтська частина побудована як SPA (Single Page Application) на основі React. Інтерфейс розділений на такі ключові компоненти:
Панель навігації (Sidebar): список модулів курсу з індикаторами прогресу (відсоток виконання) та позначками завершених тем.
Модуль теорії (TheoryViewer): відображає навчальний матеріал у структурованому вигляді з підтримкою формул (MathJax), схем та інфографіки.
Модуль завдань (TaskRunner): інтерактивне середовище для виконання практичних вправ. Залежно від типу задачі відображає різні елементи введення: таблицю для матриць МАІ, конструктор дерева цілей із можливістю drag-and-drop, форму для розрахунків.
Результати та зворотний зв'язок (FeedbackPanel): після перевірки відображає результат (вірно / частково вірно / невірно), кількість отриманих балів, правильне рішення та методичне пояснення помилки.
Дашборд студента (StudentDashboard): персональна сторінка із загальним прогресом, графіком динаміки балів, списком отриманих досягнень та рекомендаціями щодо подальшого навчання.
Адміністративна панель викладача (TeacherPanel): таблиця успішності групи, детальна статистика по кожному завданню (відсоток вірних відповідей, середня кількість спроб), інтерфейс для додавання та редагування завдань.
Особлива увага приділена ергономіці інтерфейсу. Навчальний простір максимально вільний від зайвих елементів: студент бачить умову задачі та поле для введення відповіді, а всі допоміжні елементи (підказки, прогрес, навігація) виведені на другий план. Кольорова схема обрана нейтральна, що не викликає зорового стресу при тривалій роботі.

Методика використання тренажеру в дистанційному навчанні
Ефективність навчального тренажеру визначається не лише якістю технічної реалізації, але й методичною концепцією його включення в освітній процес. Для дистанційного курсу «Системний аналіз» тренажер проєктується як обов'язкова практична компонента, що органічно доповнює теоретичні матеріали та замінює традиційні практичні заняття.
Методика використання тренажеру базується на принципі «Вивчи — Практикуй — Перевір» (Learn — Practice — Assess), реалізованому для кожного тематичного блоку курсу:
Фаза 1 — «Вивчи» (Learn): студент опрацьовує теоретичний матеріал модуля через вбудований перегляд у тренажері. Матеріал структурований за принципом від загального до часткового: спочатку концептуальний огляд, потім детальний виклад із прикладами та визначеннями. Для кожної теми передбачені вбудовані питання для самоперевірки розуміння (comprehension checks) — короткі питання після ключових концепцій.
Фаза 2 — «Практикуй» (Practice): студент виконує серію практичних завдань із поступово зростаючою складністю. На цьому етапі система надає необмежену кількість спроб, підказки після кожної помилки та демонстраційне рішення після третьої невдалої спроби. Мета цієї фази — не перевірка знань, а їх формування через активне вирішення задач.
Фаза 3 — «Перевір» (Assess): підсумкова оцінювальна сесія з обмеженою кількістю спроб та часовим лімітом. Задачі цієї фази відрізняються від практичних завдань та мають контрольний характер. Результати цієї фази автоматично передаються до журналу успішності викладача.
Рекомендована схема роботи з тренажером протягом тижня:
    • Понеділок–вівторок: опрацювання теоретичного матеріалу нового модуля (фаза «Вивчи»).
    • Середа–четвер: виконання практичних завдань (фаза «Практикуй»), не менше ніж 5 задач.
    • П'ятниця: проходження контрольного тесту модуля (фаза «Перевір»).
Ця схема реалізує принцип «розподіленої практики» (spaced practice) — один із найбільш ефективних, науково обґрунтованих методів запам'ятовування складного матеріалу. Регулярне повернення до матеріалу через певні інтервали суттєво підвищує рівень довготривалого засвоєння знань порівняно з «зубрінням» напередодні іспиту.
Роль викладача в методиці використання тренажеру:
Хоча тренажер реалізує значний обсяг автоматизованого оцінювання, роль викладача залишається ключовою. Викладач відповідає за: встановлення графіку проходження модулів і термінів здачі контрольних тестів; щотижневий аналіз статистики успішності групи для виявлення проблемних тем; проведення онлайн-консультацій для опрацювання типових помилок, виявлених тренажером; оновлення бази завдань та актуалізацію навчальних матеріалів.
3.2. Сценарії та рівні навчальних завдань
Система завдань тренажеру побудована на основі трирівневої таксономії складності, що узгоджується з ревізованою таксономією Блума для когнітивних цілей навчання. Такий підхід забезпечує поступове формування компетентностей студента — від базового розуміння до здатності застосовувати знання в нестандартних ситуаціях.
Рівень 1 — Репродуктивний (базовий):
Завдання цього рівня перевіряють розуміння термінів, визначень та базових концепцій системного аналізу. Типові формати: вибір правильної відповіді з варіантів, встановлення відповідності між поняттями, заповнення пропуску у визначенні.
Приклад завдання рівня 1 (тема «Теорія систем»):
«Оберіть правильне визначення поняття «системна властивість емерджентності»:
a) Здатність системи зберігати структуру при зміні зовнішнього середовища.
b) Наявність у системи властивостей, яких немає у жодного з її елементів окремо.
c) Можливість системи відтворювати себе у нових умовах.
d) Залежність поведінки системи виключно від властивостей її елементів.»
(Правильна відповідь: b)
Рівень 2 — Аналітичний (середній):
Завдання цього рівня вимагають від студента застосування методів системного аналізу до конкретної ситуації. Студент не просто відтворює визначення, а здійснює розрахунки, будує структури або аналізує надану модель.
Приклад завдання рівня 2 (тема «Метод аналізу ієрархій»):
«Для вибору між трьома альтернативними IT-системами за критерієм «надійність» задано матрицю попарного порівняння. Розрахуйте нормований вектор пріоритетів та індекс узгодженості (CR). Зробіть висновок про узгодженість суджень.»
(Студент вводить матрицю в табличний компонент тренажеру, система розраховує результати та порівнює їх з еталонними значеннями з урахуванням похибки ±0.01)
Рівень 3 — Творчий (підвищений):
Завдання цього рівня моделюють реальні прикладні ситуації, де студент повинен самостійно визначити підхід до вирішення задачі, побудувати модель системи та обґрунтувати прийняте рішення.
Приклад завдання рівня 3 (комплексна тема):
«Вам необхідно вибрати систему управління навчальним процесом (LMS) для університету. Наявні альтернативи: Moodle, Canvas та власна розробка. Критерії вибору: вартість впровадження, функціональність, простота використання, безпека даних, масштабованість.
Виконайте:
    1. Побудуйте ієрархію цілей для задачі вибору.
    2. Визначте відносну вагу кожного критерію методом МАІ.
    3. Оцініть альтернативи за кожним критерієм.
    4. Розрахуйте інтегральні пріоритети альтернатив та сформулюйте рекомендацію.»
(Тренажер перевіряє коректність побудованої ієрархії, правильність матриць попарного порівняння та відповідність кінцевих пріоритетів. Обґрунтування рішення передається викладачу для ручної перевірки.)
Розподіл завдань за рівнями складності:
Для забезпечення оптимального навчального ефекту рекомендований розподіл завдань у базі тренажеру: 30% — рівень 1 (репродуктивний), 50% — рівень 2 (аналітичний), 20% — рівень 3 (творчий). Такий розподіл відповідає принципу «зони найближчого розвитку» та поступово виводить студента за межі репродуктивного засвоєння.
3.3. Критерії оцінювання результатів навчання
Критерії оцінювання у тренажері виконують подвійну функцію: з одного боку, вони визначають алгоритм автоматичної перевірки відповідей, з іншого — формують прозору систему мотивації для студентів. Чіткі та зрозумілі критерії є необхідною умовою для того, щоб студент розумів, чого від нього очікують, і міг самостійно оцінювати власний прогрес.
1. Функціональні критерії якості тренажеру:
    • Тренажер коректно ідентифікує правильні та неправильні відповіді для всіх типів завдань (тести, розрахункові задачі, задачі на побудову схем).
    • Алгоритм перевірки задач на МАІ правильно розраховує вектор пріоритетів та визначає рівень узгодженості суджень з точністю ±0.01.
    • Система коректно відстежує прогрес студента та зберігає повну історію спроб.
    • Адміністративна панель забезпечує перегляд індивідуальних результатів кожного студента з деталізацією до рівня окремого завдання.
    • Механізм геймифікації (бали, досягнення, рейтинг) функціонує без помилок і коректно нараховує бонуси.
2. Система балів для оцінювання студентів:
Тип завдання	Рівень 1	Рівень 2	Рівень 3
Тест (вибір відповіді)	5 балів	10 балів	—
Розрахункова задача	10 балів	20 балів	35 балів
Задача на побудову моделі	—	25 балів	40 балів
За першу спробу нараховується повна кількість балів. За другу спробу — 70% від максимуму. За третю та подальші спроби — 50% від максимуму. Якщо студент скористався підказкою — 80% від балів за поточну спробу.
3. Критерії завершення модуля:
Модуль вважається успішно завершеним за одночасного виконання таких умов:
    • Опрацьовано не менше 80% теоретичного матеріалу модуля.
    • Виконано не менше 70% практичних завдань фази «Практикуй».
    • Контрольний тест (фаза «Перевір») пройдено з результатом не менше 60% від максимально можливого балу.
4. Критерії успішного завершення курсу:
    • Усі 5 тематичних модулів мають статус «Завершено».
    • Загальний рейтинговий бал за курс становить не менше 60% від максимуму.
    • Виконано принаймні одне завдання рівня 3 (творчого) у кожному модулі.
5. Технічні критерії якості системи:
    • Відсутні критичні помилки (помилки рівня Critical та High).
    • Час відгуку серверу при перевірці відповіді — не більше 2 секунд.
    • Система коректно працює в браузерах Chrome, Firefox, Safari та Edge останніх двох версій.
    • Автентифікація та розмежування доступу між ролями (студент / викладач / адміністратор) функціонують без збоїв.
    • Механізм резервного копіювання бази даних виконується щодобово без помилок.
